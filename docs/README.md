# OctoAcme Project Management Docs

## Overview

This folder contains the comprehensive set of process documents that guide how OctoAcme runs projects. Our approach is built on principles of customer-first delivery, iterative development, clear ownership, and data-informed decisions. These living documents capture our collective wisdom on how to execute projects consistently, transparently, and with measurable outcomes.

## Core Principles

- **Customer-first**: Prioritize customer value and usability in every decision
- **Iterative delivery**: Deliver small, testable increments and validate assumptions early
- **Clear ownership**: Each project has named Project Manager and Product Lead with accountability
- **Data-informed decisions**: Measure impact and iterate based on evidence, not intuition
- **Psychological safety**: Encourage feedback, learning, and continuous improvement

## Process Documents

### [Project Management Overview](octoacme-project-management-overview.md)
Introduction to OctoAcme's project management framework, core roles (PM, PdM, Developers, QA), key artifacts, and the project lifecycle (Initiation → Planning → Execution → Release → Close).

### [Project Initiation Guide](octoacme-project-initiation.md)
Guidance for validating and authorizing new work, aligning stakeholders, and creating the initial project plan. Includes the Project One-pager template and a go/no-go decision gate.

### [Project Planning](octoacme-project-planning.md)
Steps to turn an approved initiative into an actionable plan and backlog for delivery. Covers estimation, Definition of Done, dependency identification, and release planning.

### [Execution & Tracking](octoacme-execution-and-tracking.md)
Day-to-day guidance for team rhythm (standups, syncs, demos), PR workflow, quality standards, testing requirements, and a three-level blocker escalation process.

### [Risks & Communication](octoacme-risks-and-communication.md)
How to identify, assess, and manage risks via a Risk Register. Includes stakeholder communication templates (weekly status, incident communication) and escalation paths.

### [Release & Deployment](octoacme-release-and-deployment.md)
Standardized process for releasing to production. Covers release types, pre-release requirements, deployment checklists, smoke tests, rollback procedures, and incident playbooks.

### [Retrospective & Continuous Improvement](octoacme-retrospective-and-continuous-improvement.md)
How to run effective retrospectives, capture learnings, and convert them into actionable improvements with clear owners and timelines.

### [Roles & Personas](octoacme-roles-and-personas.md)
Definitions of typical project roles (Project Manager, Product Manager, Developer) including their responsibilities, goals, and communication patterns.

## Getting Started

**New to OctoAcme?**
- Start with [Project Management Overview](octoacme-project-management-overview.md) for a high-level introduction to our framework and roles.

**Initiating a new project?**
- Follow the [Project Initiation Guide](octoacme-project-initiation.md) to validate the opportunity and create your One-pager.

**Planning or executing a project?**
- Reference [Project Planning](octoacme-project-planning.md) or [Execution & Tracking](octoacme-execution-and-tracking.md) for your current phase.

**Managing risks or communicating status?**
- Check [Risks & Communication](octoacme-risks-and-communication.md) for templates and escalation guidance.

**Preparing a release?**
- Use [Release & Deployment](octoacme-release-and-deployment.md) for standardized procedures and checklists.

**Finishing a project or sprint?**
- Run a retrospective using [Retrospective & Continuous Improvement](octoacme-retrospective-and-continuous-improvement.md).

## How to Use These Docs

- **Keep current**: Update process docs as workflows evolve and improvements are validated.
- **Add project charters**: Store project One-pagers and plans in your project repository (not here).
- **Copilot Spaces integration**: Add process-specific docs to `.copilot/` if you want Copilot Spaces to use them as context for assistance.
- **Use checklists**: Leverage checklists as quality gates and progress markers during projects.
- **Continuous improvement**: Add action items to your project backlog or GitHub Issues when you identify improvements to these processes.
